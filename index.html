01 — Objective

เป้าหมายและคำถามที่ต้องตอบ

04 — Scoring Model

เกณฑ์การให้คะแนน (100 คะแนน)

# หมวดคะแนน น้ำหนัก สัดส่วน เกณฑ์การพิจารณา
1 Rating & Review Quality
คุณภาพคะแนนและรีวิว		 25
Google ≥4.3 = เต็ม, จำนวนรีวิว ≥500 บวก 5pt, Michelin = บวก 5pt
2 Group Suitability
ความเหมาะกับกลุ่ม		 20
รองรับ 10+ คน, มีห้องส่วนตัว/โต๊ะใหญ่, เหมาะบรรยากาศทีม
3 Price Suitability
ความเหมาะของราคา		 15
200–500 บาท/คน = เต็ม, <200 = minus 3pt (เล็กไป), >800 = minus 5pt
4 Travel Convenience
ความสะดวกการเดินทาง		 15
เดิน <500m จาก BTS อารีย์ = เต็ม, มีที่จอดรถ = บวก 3pt
5 Data Completeness
ความครบของข้อมูล		 15
ครบ 11 field = เต็ม, ขาด field ละ 1pt, มี source URL = บวก 2pt
6 Uniqueness / Experience
ความพิเศษ/ประสบการณ์		 10
มีประสบการณ์พิเศษ (live station, open kitchen, concept), ไม่ซ้ำกับร้านอื่น
รวมทั้งหมด 100 คะแนนสูงสุดเป็นผู้แนะนำ (ห้ามเปลี่ยนน้ำหนัก)

08 — Short Reflection

สรุปการเรียนรู้จากโปรเจกต์

4.1 ขั้นตอนทำจริงเป็นอย่างไร

เริ่มจากการตั้งโจทย์ให้ชัดเจนว่าต้องการหาร้านสำหรับทีม 8–12 คน ย่านอารีย์ โดยใช้ Apify Google Maps Scraper ดึงข้อมูลร้านย่านอารีย์ ได้ raw data 50 ร้าน แล้วคัดเฉพาะร้านในย่านอารีย์ที่เหมาะกลุ่ม 22 ร้าน จากนั้นนำเข้า Google Sheets แบ่ง tab raw/clean/scoring → ใช้ Claude prompt เพื่อ normalize ข้อมูล (ราคา, ประเภทอาหาร, สถานะ group suitability) → ให้คะแนนตาม scoring model → ต่อ n8n workflow อัตโนมัติเชื่อม Sheets กับ Claude API เพื่อสร้าง summary text → สุดท้ายสร้าง HTML รายงานหน้านี้

4.2 จุดที่ทำได้ดี และจุดที่ติดปัญหา

ทำได้ดี: Apify ดึงข้อมูล Google Maps ได้ครบและแม่นยำ ได้ทั้ง rating, review count, ที่อยู่จริง — ไม่ต้องกรอกเอง ประหยัดเวลาได้มาก การใช้ Claude ช่วย normalize ข้อมูล (เช่น "฿฿฿" → "400–600฿/คน") ทำให้ข้อมูลเทียบกันได้ทันที

ติดปัญหา + AI แนะนำผิด: ตอนแรก AI ช่วยกรอก rating/review โดยประมาณจากบทความเว็บ ทำให้ตัวเลขคลาดเคลื่อน (เช่น เซนิกุเทนกรอก 4.4/3,363 แต่ Apify จริง = 4.5/1,511) — แก้โดยรัน Apify Google Maps Scraper ดึงเลขจริง 50 ร้าน แล้วนำมา cross-check และอัปเดตทุกค่าให้ตรงหลักฐาน | n8n connection กับ Claude API ครั้งแรก timeout เพราะ max_tokens ตั้งต่ำเกินไป แก้โดยเพิ่มเป็น 1024 | บางร้าน "Group Suitability" ไม่ชัดเจนใน web → ใช้ Claude inference จาก review/ประเภทร้านแทน

4.3 สิ่งที่จะนำไปปรับปรุงในงานชิ้นต่อไป

1. เพิ่ม data source เช่น Yelp หรือ Facebook reviews เพื่อ cross-validate — ป้องกัน bias จากแหล่งเดียว
2. Scoring automation ให้ครบ — ตอนนี้บางหมวดยังให้คะแนนแบบ manual ควร map เป็น formula ใน Sheets
3. n8n scheduling — ตั้ง trigger อัตโนมัติทุกสัปดาห์ไป re-scrape ข้อมูลใหม่ เพราะร้านอาหารเปิด-ปิดบ่อย
4. User input — เพิ่ม filter หน้า HTML ให้ผู้ใช้กรองตามงบ, ประเภทอาหาร, จำนวนคนได้แบบ real-time
